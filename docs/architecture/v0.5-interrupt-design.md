# Design: v0.5 Asynchronous Interrupt Handler (Conceptual)

## Overview
The Asynchronous Interrupt Handler is the bridge between the real-time sensor edge and the sequential cognitive kernel. It ensures that external signals can gracefully preempt or buffer without causing state corruption or data loss.

## 1. The Interrupt Signal (Input Structure)
```python
{
    "sensor": "USER_DIRECT",
    "priority": 10,
    "stimulus": "ABORT current task and search for X",
    "policy": "INTERRUPT"
}
```

## 2. Kernel Atomicity (The Step Boundary)
The `Executor` will be updated with a check-in mechanism.
- **Rule**: No interruption occurs during a tool execution.
- **Mechanism**: The `Engine` checks a `_pending_interrupt` flag at the start of every competition loop or after every `Executor.execute()` call.

## 3. Context Serialization (The Snapshot)
When an INTERRUPT is accepted, the `Engine` performs a **System Snapshot**:
- `trace_id` of the interrupted task.
- `last_known_score` and `total_steps`.
- Current procedural pattern being attempted.
- `working_memory` contents.
This is persisted to `memory/episodic.json` with a `STATUS: PAUSED` marker.

## 4. Preemption Logic (The Decision Tree)
The `Engine` invokes `_evaluate_effort()` with the new stimulus.
- **If Priority(New) > Priority(Current)**:
    - Current Task -> `Buffer / Pause`
    - New Task -> `Execute`
- **If Priority(New) <= Priority(Current)**:
    - New Task -> `Buffer`
    - Current Task -> `Continue`

## 5. Resumption Protocol
After the interrupting task completes (or reaches a plateau):
1. `Engine` scans `memory/episodic.json` for `STATUS: PAUSED` traces.
2. The `Planner` re-hydrates the saved context.
3. The competition loop resumes from the next available strategy run.

## Invariant Check
> **No Sensor directly calls handle_interrupt.**
All sensors post to a shared **Signal Buffer**. The `Engine` polling cycle is the *only* entry point for preemption.
