## Summary

Seven fixes targeting correctness, security, and memory safety:

### 🔴 Critical: cortex.py — O(N²) memory blowup in sleep_cycle
The full N×N cosine similarity matrix for 100K+ entries would consume ~40GB. Replaced with chunked row-grouping: peak memory drops from O(N²) to O(N×dim×chunk) (~0.8GB per chunk at CHUNK=1024).

### 🔴 Critical: security.py — Timing attack vulnerability
`_verify_sudo_password` used `==` for hash comparison. Changed to `hmac.compare_digest` for constant-time verification.

### 🟡 High: model.py — Thread count leak on exception
`torch.set_num_threads()` was not guarded by try/finally. If mesh.forward() threw an exception, the thread count would remain changed for the rest of the process. Now properly restored in `finally`.

### 🟡 High: generation.py — last_prompt_len inconsistency
When `encode()` returns empty list, `last_prompt_len` was 0 but the BOS token was injected into `ids`, causing `get_routing_telemetry` to misclassify the BOS as generated output.

### 🟡 High: model_failover.py — streaming_completion no timeout/error handling
Subprocess stderr was never read (deadlock risk), no timeout on streaming (infinite hang), no return-code checking. Added background stderr drain, asyncio.wait_for for timeouts, and exit code verification.

### 🟢 Medium: context.py — process_invalidation was dead code
The method cleared the queue without processing entries. Now iterates and evicts each pending key.

### 🟢 Medium: plasticity.py — auto-scaler prune_cortex action never executed
`PlasticityAutoScaler.check_and_scale()` returned `"prune_cortex"`, `"add_strand"`, `"add_layer"` actions but the `PlasticityEngine.check()` method never called the autoscaler. Now wired into the check cycle.
